Supervisor Brain - Mimari Yaklaşımlar

Netleşen kararlar doğrultusunda 3 yaklaşım karşılaştırması

Onaylanan Kararlar
1. Full Autonomous Team Lead
2. Tek provider: aynı pipeline + role-based verify
3. Adaptive activation (complexity threshold)
4. Hybrid capability analysis (heuristic + cheap LLM)
5. GoalDecomposer dokunulmaz, arada katmanlar
6. Chat: minimal progress / Monitor: full detail
A

Full Pipeline (Önerilen)

5 yeni modül, tam pipeline, tüm kararları implemente eder

User Request
|
TaskClassifier
|-- simple/moderate --> Mevcut PAOR (sıfır overhead)
|-- complex -->
|
SupervisorBrain (yeni orkestratör)
|
GoalDecomposer (mevcut, dokunulmaz)
| DAG nodes
CapabilityMatcher (yeni)
| capability tags per node
ProviderAssigner (yeni)
| provider:model per node
SupervisorDispatcher (yeni)
| wave-based parallel PAOR loops
ResultAggregator (yeni)
| cross-validate + merge
User Response

Artılar

  • Tüm kararları tam implemente eder
  • Her modül bağımsız test edilebilir
  • Clean separation of concerns
  • Tek ve çoklu provider aynı pipeline
  • Monitor panel için zengin telemetry

Eksiler

  • 5 yeni modül = daha fazla kod
  • Pipeline overhead (ama sadece complex task'larda)
  • Integration testing karmaşıklığı

Artılar

  • Daha az modül, daha hızlı implemente
  • Daha az wiring karmaşıklığı
  • Yeterince modüler

Eksiler

  • Match + Assign birleşik = test zorlaşır
  • Dispatch + Aggregate birleşik = büyüyebilir
  • İleride ayırmak gerekebilir
C

Incremental (Fazlı Geliştirme)

Önce core, sonra genişlet - 2 fazda delivery

Faz 1: Temel Supervisor
GoalDecomposer --> CapabilityMatcher
--> ProviderAssigner --> mevcut GoalExecutor
(mevcut execution, yeni routing)
 
Faz 2: Tam Pipeline
+ SupervisorDispatcher (paralel PAOR)
+ ResultAggregator (cross-validate)
+ Monitor panel entegrasyonu

Artılar

  • Erken değer: Faz 1 ile hemen provider routing kazanılır
  • Risk azaltma: adım adım
  • Faz 1 sonrası feedback alıp Faz 2'yi şekillendir

Eksiler

  • Faz 1 tam vizyonu deliver etmez
  • Geçici mimari kararlar gerekebilir
  • İki ayrı integration cycle
Önerim: A (Full Pipeline)
B yüzeysel tasarruf sağlar ama birleşik modüller büyüyünce yine ayırmak gerekir (orchestrator'da yaşadık). C mantıklı görünür ama Faz 1'in geçici mimari kararları Faz 2'yi zorlaştırabilir. A baştan doğru modüler yapıyı kurar — her modül 150-300 satır, focused, test edilebilir. Zaten mevcut GoalDecomposer + GoalExecutor + GoalValidator + GoalStorage pattern'ını takip eder.
