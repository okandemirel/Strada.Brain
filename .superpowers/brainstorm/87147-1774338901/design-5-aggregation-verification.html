Conflict Resolution - Node Sonuçları Arasında Çelişki
Ne Zaman Çelişki Olur?
1. Node A "users" tablosu oluşturur, Node B farklı schema bekler
2. Node C bir fonksiyon yazar, Node D aynı isimde farklı fonksiyon
3. Node E bir config değiştirir, Node F aynı config'i farklı değiştirir
4. Verification "flag_issues" döner, orijinal node re-execute gerekir
Nasıl Çözülür?
1. File-level conflict detection: aynı dosyayı değiştiren node'lar tespit et
2. LLM-driven merge: çelişen değişiklikleri LLM ile birleştir
3. Dependency ordering: DAG sırasına göre sonraki node'un değişikliği kazanır
4. Fallback: otomatik çözülemezse kullanıcıya sun

Verification Konfigürasyonu
interface VerificationConfig {
  mode: "always" | "critical-only" | "sampling" | "disabled";
  samplingRate: number; // 0-1, sampling modunda verify oranı
  preferDifferentProvider: boolean; // true: farklı provider ile verify
  maxVerificationCost: number; // toplam verification maliyeti limiti
}
 
// Defaults:
// mode: "critical-only" (sadece quality preference'lı node'ları verify)
// samplingRate: 0.3 (sampling modunda %30)
// preferDifferentProvider: true
// maxVerificationCost: agent budget'ın %15'i
